### The Generic Naming Problem

February's 63% error rate (vs January's 23%) reveals a **critical issue with generic file names**:

**High-Risk Prefixes (Avoid in Future):**
| Prefix | Total Files | Error Rate | Recommendation |
|--------|------------|-----------|-----------------|
| `screenshot-` | 15 | 100% | ❌ NEVER use - always specify platform |
| `email-[fragment]` | 18 | 89% | ❌ NEVER use generic email fragments |
| `automation-automation-rule-` | 8 | 100% | ❌ NEVER use - too vague |
| `calendar-meeting-` | 1 | 100% | ❌ Verify platform before naming |
| `code-` | 2 | 100% | ❌ Usually documents, not code |

**Low-Risk Prefixes (Safe to Use):**
| Prefix | Total Files | Error Rate | Recommendation |
|--------|------------|-----------|-----------------|

| `spreadsheet-` | 5 | 0% | ✅ Always accurate |
| `email-[PERSON]` | 4 | 0% | ✅ Specific names are reliable |
| `document-[SPECIFIC]` | 2+ | 0% | ✅ Named documents accurate |
| `browser-` | 2 | 0% | ✅ Always accurate |

### Why February Had Higher Errors

1. **Larger proportion of generic names** (33/81 files = 41% generic vs January's ~25%)
2. **More HubSpot CRM records** mislabeled as email (18 files)
3. **Bulk "screenshot-" prefix** with no differentiation (15 files)
4. **More automation-rule artifacts** from workflow testing (8 files)

### Lessons Learned

✅ **What Works:**
- Specific platform names: `slack-`, `spreadsheet-`, `browser-`
- Named contacts: `email-[PERSON]`, `document-[TOPIC]`
- Platform + detail: `website-[SERVICE]-[DETAIL]`, `document-[TYPE]-[SUBJECT]`

❌ **What Doesn't Work:**
- Single-word prefixes without context: `email-`, `document-`, `screenshot-`
- Workflow artifact names: `automation-automation-rule-`
- Time fragments: `email-date:`, `email-get-`, `email-@-`
- Generic descriptions: `calendar-meeting-`, `code-`
